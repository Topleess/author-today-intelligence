# Методика поиска вероятных незаконных публикаций книг

Статус: проект безопасного discovery-процесса для concierge-MVP.  
Дата: 2026-07-17.

## 1. Задача

Найти публичные материалы, которые технически похожи на произведение клиента, не называя их нарушением до проверки контекста, разрешений и правовой квалификации.

Конвейер:

```text
правообладатель и произведение
→ контрольные признаки
→ разрешённые поисковые поверхности
→ кандидаты
→ техническое сопоставление
→ дедупликация
→ проверка контекста и разрешений
→ ручная квалификация
→ доказательный отчёт
```

## 2. Входной пакет

До поиска клиент предоставляет:

- название произведения и варианты написания;
- имя, псевдонимы и соавторов;
- официальную страницу оригинала;
- статус произведения и дату первой публикации;
- файл или текст, который клиент вправе предоставить для сопоставления;
- перечень разрешённых площадок, издательств, магазинов и партнёров;
- известные официальные аудиоверсии;
- старые названия, редакции и обложки;
- уже известные подозрительные URL;
- подтверждение авторства/правообладания либо полномочий представителя.

Без списка разрешённых размещений качество фильтрации будет заведомо ниже.

## 3. Контрольные признаки

### Текстовые

Для одной книги формируется 10–20 поисковых фрагментов:

- 8–14 слов подряд;
- редкая комбинация имён, мест и действий;
- характерная реплика;
- фрагмент из начала, середины и конца;
- фрагменты из разных глав;
- намеренно исключаются типовые фразы и общеупотребительные выражения.

Для каждого фрагмента сохраняются:

```text
fragment_id
work_id
chapter_or_location
normalized_text
source_hash
character_count
word_count
search_priority
created_at
```

Полные рукописи не передаются поисковым системам. Во внешний запрос отправляется только минимальный необходимый фрагмент.

### Метаданные

- точное название;
- название + автор;
- название + «читать онлайн»;
- название + «скачать»;
- название + форматы файлов;
- название + «аудиокнига»;
- название + номер главы;
- названия серии и тома;
- имена редких персонажей;
- старое название или вариант транслитерации.

Такие запросы являются способом обнаружения кандидатов, а не доказательством незаконности.

### Визуальные

Для обложек можно использовать:

- перцептивный хеш;
- уменьшенную локальную копию;
- поиск точных/похожих изображений через разрешённый интерфейс;
- OCR названия и имени автора.

Совпадение обложки не доказывает размещение текста и хранится как отдельный слабый сигнал.

### Аудио

Для официальной аудиоверсии возможны:

- сравнение коротких аудиофингерпринтов;
- распознавание короткого публичного фрагмента и текстовое сопоставление;
- сверка длительности, названия глав, диктора и обложки;
- ручная прослушка минимального участка.

Не скачивать закрытые или платные аудиофайлы и не обходить ограничения доступа.

## 4. Источники и режимы доступа

Каждый источник до подключения получает карточку:

| Поле | Содержание |
|---|---|
| Источник | название и домен |
| Способ доступа | официальный API / обычный ручной поиск / пользовательский экспорт |
| Условия | ссылка и дата проверки |
| Автоматизация | разрешена / ограничена / запрещена / неясно |
| Лимит | документированный предел или внутренний безопасный предел |
| Хранение | какие данные и на какой срок можно сохранять |
| Авторизация | не нужна / только человек / интеграция по официальному OAuth/API |
| Stop-condition | CAPTCHA, запрет правил, платный доступ, блокировка, неясный статус |

### Допустимые поверхности первого пилота

1. Обычный ручной веб-поиск.
2. Официальные поисковые API при наличии подходящего тарифа и разрешения на использование результатов.
3. Публичные страницы, выбранные человеком.
4. Пользовательские сообщения о конкретных URL.
5. Точечная проверка архивного снимка конкретного уже известного URL.
6. Официальные формы жалоб и правообладательские кабинеты, которые использует человек.

### Что не входит

- массовый обход доменов;
- скрытые внутренние API без разрешения;
- автоматизация под чужой сессией;
- обход CAPTCHA, paywall или регистрации;
- сбор личных переписок и закрытых групп;
- эксплуатация утечек и пиратских баз как источника данных;
- повторная публикация полного найденного текста;
- автоматическая рассылка требований.

## 5. Генерация запросов

Для каждого произведения создаются группы запросов.

### Группа A: точные фрагменты

```text
"[редкий фрагмент 8–14 слов]"
"[редкая реплика]"
"[фрагмент]" "[имя персонажа]"
```

### Группа B: название и формат

```text
"[название]" "[автор]"
"[название]" читать онлайн
"[название]" скачать
"[название]" epub
"[название]" fb2
"[название]" pdf
"[название]" аудиокнига
```

### Группа C: структура произведения

```text
"[название]" "глава [номер]"
"[серия]" "том [номер]"
"[редкий персонаж]" "[редкое место]"
```

### Группа D: известный домен

Только после ручного обнаружения домена:

```text
site:[домен] "[название]"
site:[домен] "[редкий фрагмент]"
```

Это ограниченный поиск через индекс, а не разрешение сканировать сам домен.

## 6. Нормализация кандидата

Для каждого результата сохраняется минимум:

```text
candidate_id
query_id
result_url
canonical_url
observed_title
observed_snippet
source_provider
source_rank
found_at
access_class
collection_method
```

Нормализация URL:

- только `http`/`https`;
- удалить известные tracking-параметры;
- сохранить исходный URL отдельно;
- привести домен к каноническому виду;
- не переходить по сокращённой ссылке автоматически, если назначение не проверено;
- не следовать редиректам на иной домен без отдельной фиксации;
- не принимать URL с учётными данными или нестандартным портом без ручной проверки.

## 7. Техническое сопоставление текста

Порядок от наиболее надёжного к исследовательскому:

1. точное совпадение контрольного фрагмента;
2. совпадение после нормализации пробелов, кавычек и дефисов;
3. локальное шинглирование по словам;
4. локальная метрика редакционного сходства;
5. модельная оценка как вспомогательный сигнал с обязательным указанием цитат-доказательств.

Модель не получает право выставлять `likely_infringement`. Она может только предложить кандидата и объяснить наблюдаемое сходство.

### Минимальные технические метрики

```text
exact_match_count
matched_word_count
longest_contiguous_match
normalized_similarity
matched_locations
comparison_version
```

Порог отбора к ручной проверке задаётся после пилота на реальных положительных и отрицательных примерах. До этого нельзя объявлять универсальный процент, который означает нарушение.

## 8. Дедупликация

Один материал может встречаться в зеркалах, агрегаторах и нескольких URL.

Группировать по:

- каноническому URL;
- домену и нормализованному пути;
- хешу доступного контента;
- совпадающим фрагментам;
- названию файла и размеру;
- ссылке на один и тот же источник/хранилище.

В отчёте показывать один основной кейс и список зеркал. Не продавать количество дублей как количество отдельных нарушений.

## 9. Ручная фильтрация

Оператор последовательно проверяет:

1. Это действительно произведение клиента?
2. Площадка или аккаунт есть в allowlist разрешённых размещений?
3. На странице находится текст/аудио или только описание и ссылка?
4. Какой объём материала доступен?
5. Указаны ли автор и источник?
6. Какова заявленная цель публикации: рецензия, обсуждение, обучение, продажа, бесплатное чтение?
7. Есть ли признаки официального магазина, библиотеки, издателя или партнёра?
8. Достаточно ли данных для проверки или статус должен остаться `unclear`?

Результат фиксируется одним из классов:

- `authorized`;
- `reference_only`;
- `quotation_review`;
- `likely_infringement`;
- `unclear`.

## 10. Очередь приоритетов

Приоритет не равен юридической виновности.

### Высокий

- доступен полный текст, несколько глав, файл или полная аудиоверсия;
- материал предлагается за деньги либо используется для монетизации;
- высокая посещаемость/охват подтверждается наблюдаемыми данными;
- контент может быстро исчезнуть;
- совпадение подтверждено несколькими независимыми признаками.

### Средний

- существенный фрагмент, но объём и контекст требуют оценки;
- небольшое зеркало полного материала;
- источник или разрешение неясны;
- доказательства доступны и риск исчезновения невысок.

### Наблюдение

- упоминание, обложка, короткий фрагмент;
- слабое техническое совпадение;
- страница недоступна;
- нет достаточных данных о фактическом содержании.

## 11. Контроль качества

Для каждого пилота считать:

```text
queries_run
raw_candidates
unique_candidates
false_positive_count
authorized_count
reference_only_count
quotation_review_count
likely_infringement_count
unclear_count
human_review_minutes
capture_minutes
cost_per_reviewed_candidate
cost_per_confirmed_case
```

Ключевые показатели продукта:

- доля кандидатов, переживших ручную фильтрацию;
- время на один проверенный кейс;
- число ранее неизвестных автору значимых случаев;
- принятие отчёта автором/юристом;
- готовность платить за повторный мониторинг.

## 12. Первая проверка механизма

До разработки автоматического коннектора:

1. Выбрать одну книгу с разрешения правообладателя.
2. Получить 10 контрольных фрагментов и allowlist официальных размещений.
3. Вручную выполнить 20–30 запросов.
4. Сохранить только кандидатов и метаданные, необходимые для проверки.
5. Заполнить отчёт по максимум пяти различным кейсам.
6. Замерить трудозатраты по этапам.
7. Передать спорные случаи юристу без заранее заданного вывода.
8. Решить, какой один этап стоит автоматизировать следующим.

Первый кандидат на автоматизацию — локальная нормализация, дедупликация и заполнение карточки. Не массовый веб-сбор и не отправка претензий.
